## 验证方法论

### 验证层次

1. **编译/构建通过**：代码无编译错误和新增 warning。
2. **单元验证**：核心逻辑的输入输出是否符合预期。
3. **集成验证**：修改的模块与上下游模块的交互是否正常。
4. **场景验证**：在实际运行环境（Editor Play Mode / 真机）中复现目标行为。
5. **回归验证**：原有功能是否仍然正常，特别是与修改代码共享路径的功能。

### 验证策略选择

| 变更类型 | 最低验证要求 |
|----------|------------|
| 单文件 bug 修复 | 编译 + 复现路径验证 + 回归 |
| 模块内功能新增 | 编译 + 功能验证 + 接口调用方验证 |
| 跨模块重构 | 编译 + 全路径验证 + 回归 + 性能对比 |
| 配置/数据变更 | 加载验证 + 运行时行为验证 |

### 验证报告格式

```
验证结果：通过 / 未通过

验证项：
  ✅ 编译通过
  ✅ {具体验证场景1} — 预期行为已确认
  ❌ {具体验证场景2} — 实际结果与预期不符
     预期：{...}
     实际：{...}

结论：{通过/需要修复/需要进一步调查}
```

### 常见遗漏

- 只验证了正常路径，没验证边界和异常路径。
- 在 Editor 验证通过就认为完成，没考虑打包后/真机上的行为差异。
- 验证环境与目标环境不一致（分辨率、平台、数据量）。
