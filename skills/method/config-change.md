## 配置变更方法论

### 变更前评估

- **影响范围**：此配置被哪些模块/流程引用？修改后谁会受影响？
- **回滚方案**：如果变更出问题，如何快速恢复？是否需要备份当前值？
- **依赖检查**：是否有其他配置与此配置联动（如资源路径与加载逻辑的对应关系）？

### 变更类型与流程

| 类型 | 流程 |
|------|------|
| **开发期调参**（如数值调优） | 直接修改 → 运行验证 → 确认效果 |
| **环境配置**（如路径、服务器地址） | 检查环境差异 → 修改 → 验证各环境一致性 |
| **框架级配置**（如 Unity ProjectSettings） | 评估影响 → 备份原值 → 修改 → 编译+全量验证 |
| **构建/发布配置**（如 Addressables 设置） | 评估影响 → 修改 → 本地构建验证 → CI 验证 |

### 安全原则

- **一次只改一项**：避免同时修改多个配置，无法判断哪个变更导致问题。
- **记录变更理由**：在提交信息中说明为什么改、改前的值是什么。
- **序列化格式一致**：JSON/YAML/XML 配置文件保持格式化风格一致，避免无关的 diff 噪音。
- **不要硬编码**：如果一个值可能被多处引用或将来需要调整，应提取为配置项而非写死在代码中。

### 热更配置的额外注意

- 确认配置变更是否需要客户端更新才能生效。
- 如果是服务器下发的配置，确认变更的灰度策略和回滚机制。
- 数值类配置注意上下限校验，防止异常值导致运行时错误。
